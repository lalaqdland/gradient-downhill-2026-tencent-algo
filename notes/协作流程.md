# 协作流程

## 角色与权限

- 仓库所有者负责邀请队友成为仓库 `collaborator`
- 对个人账号仓库来说，`collaborator` 默认拥有读写权限，可以推分支、创建 Pull Request，也可以在规则允许时执行 merge

## `main` 分支规则

建议在 GitHub 仓库设置中为 `main` 开启以下规则：

1. 必须通过 Pull Request 合并
2. 至少需要 `1` 个 approving review
3. 有新提交时自动使旧的 approval 失效
4. 所有 review conversation 必须先解决
5. 后续接入 CI 后，再把 required status checks 加上

## 日常协作约定

1. 每个人都从最新的 `main` 切出自己的工作分支
2. 在自己的分支完成提交后，发起 Pull Request 到 `main`
3. Pull Request 作者不能批准自己的 Pull Request
4. 另一位协作者负责 review、评论和 approve
5. approve 之后，仓库所有者和 collaborator 都可以执行 merge

## 建议分支命名

- `feature/<topic>`
- `fix/<topic>`
- `docs/<topic>`
- `chore/<topic>`

## 冲突处理建议

- 合并前先把目标分支同步到最新 `main`
- 如果出现冲突，由发起 Pull Request 的人优先处理
- merge 后发现问题，优先使用新的修复 Pull Request，而不是直接在 `main` 上改

## 何时由仓库所有者亲自 merge

虽然 collaborator 可以在规则允许时执行 merge，但以下情况建议由仓库所有者亲自 merge：

- 涉及最终提交材料
- 会影响目录结构或协作规范
- 同时改动多个模块，回滚成本较高
- 需要在 merge 前做最后一次人工检查
