# 协作流程

## 角色与权限

- 仓库所有者负责邀请队友成为仓库 `collaborator`
- 当前仓库只邀请 `1` 位队友作为 `collaborator`
- 对个人账号仓库来说，`collaborator` 默认拥有读写权限；在当前“不加分支保护”的设置下，他可以直接推送分支、创建 Pull Request，也可以直接推送到 `main` 或自行 merge

## GitHub 设置路径

1. 打开仓库主页，进入 `Settings`
2. 打开 `Collaborators` 或 `Manage access`
3. 邀请你的队友成为这个仓库的 `collaborator`
4. 当前不需要为 `main` 配置 branch protection rule

## 当前协作策略

当前采用更轻量的协作方式：

1. 你和队友都可以直接在仓库里工作
2. 小改动可以直接推送
3. 较大的改动、目录调整或需要讨论的内容，仍然建议走 Pull Request
4. 因为没有分支保护，Pull Request 不是强制的，队友在有需要时也可以自己 merge

## 日常协作约定

1. 开始工作前先同步最新的 `main`
2. 如果只是补资料、修文档或做很小的改动，可以直接提交
3. 如果改动较大，建议先切分支再发 Pull Request
4. 当前没有强制 review 和强制 merge 流程，是否 review 由你们自己约定
5. 如果队友直接提交到了 `main`，你本地只需要正常 `pull` 同步，不需要额外做 merge 审批

## 建议分支命名

- `feature/<topic>`
- `fix/<topic>`
- `docs/<topic>`
- `chore/<topic>`

## 冲突处理建议

- 合并前先把目标分支同步到最新 `main`
- 如果出现冲突，由发起 Pull Request 的人优先处理
- 如果有人直接推送到 `main`，另一方先 `pull` 最新版本再继续开发
- 发现问题时，优先补一个新的修复提交，而不是回退对方已经完成的工作

## 何时仍然建议由仓库所有者确认

虽然当前 collaborator 可以自己 merge 或直接推送，但以下情况建议先和仓库所有者确认：

- 涉及最终提交材料
- 会影响目录结构或协作规范
- 同时改动多个模块，回滚成本较高
- 需要在提交前做最后一次人工检查
